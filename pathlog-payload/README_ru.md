# Pathlog Monitor

`pathlog-payload` это набор утилит для:
- записи журнала обращений к файлам на PS5
- последующего разбора журнала и проверки файлов игры

Папка содержит:
- `pathlog_all.elf`  
  непрерывный мониторинг всех событий
- `pathlog_apps.elf`  
  мониторинг только во время запуска игр `CUSA*` и `PPSA*`
- `pathlog_stop.elf`  
  корректная остановка активного monitor-процесса
- `analyze_pathlog.py`  
  скрипт для анализа журнала и переименования файлов под нужный регистр

## Для чего это нужно

Проект полезен, когда нужно:
- понять, какие файлы реально читает игра
- собрать список используемых файлов
- найти ошибки регистра в именах файлов и каталогов
- сравнить журнал запуска с локальной папкой игры или с FTP-копией
- автоматически привести имена файлов и папок к регистру, который использует игра

## Что нужно перед запуском

Перед использованием должен быть загружен специальный kstuff-lite с поддержкой журналирования.  

Для сборки ELF нужен установленный SDK:

```bash
/opt/ps5-payload-sdk
```

## Быстрый старт

### Сборка

```bash
make -C pathlog-payload clean all
```

### Вариант 1: логировать всё подряд

Запусти:

```text
pathlog_all.elf
```

Что будет происходить:
- монитор сразу включит логирование
- новые события будут выводиться в консоль
- журнал будет записываться в:

```text
/data/pathlog/all.log
```

### Вариант 2: логировать только запущенные игры

Запусти:

```text
pathlog_apps.elf
```

Что будет происходить:
- monitor будет ждать запуск приложения
- когда стартует игра с кодом `CUSA*` или `PPSA*`, логирование включится автоматически
- для каждой игры создаётся отдельный журнал:

```text
/data/pathlog/<TITLE_ID>.log
```

Пример:

```text
/data/pathlog/PPSA12345.log
```

Этот режим удобен, когда нужен чистый журнал только для одной игры, без фонового шума системы.

### Остановка мониторинга

Для остановки всегда используй:

```text
pathlog_stop.elf
```

Утилита находит активный monitor-процесс по имени и завершает его корректно.  
Если монитор не запущен, она просто сообщает об этом.

## Куда пишутся журналы

Все журналы сохраняются в:

```text
/data/pathlog/
```

Основные варианты:
- `all.log` для общего мониторинга
- `<TITLE_ID>.log` для режима мониторинга игр

Формат строки журнала:

```text
timestamp<TAB>pid<TAB>kind<TAB>status<TAB>retval<TAB>arg0<TAB>arg1<TAB>path1<TAB>path2
```

Пример:

```text
2026-03-18T01:23:45.678	1337	open	ok	5	0	0	/app0/eboot.bin
2026-03-18T01:23:45.700	1337	rename	err:2	-	0	0	/app0/foo.bin	/app0/Foo.bin
```

Что означают поля:
- `pid` это процесс, который запросил syscall
- `status` это `ok` или `err:<errno>`
- `retval` это возвращённое значение syscall при успехе
- `arg0` и `arg1` это дополнительные числовые аргументы конкретного syscall
- `path2` используется для операций с двумя путями, например `rename`, `link`, `symlink`

## Выбор фильтров

Текущее встроенное поведение:
- `pathlog_all.elf` включает все поддерживаемые типы событий, без path filter и без `pid` filter
- `pathlog_apps.elf` включает все поддерживаемые типы событий, ставит kernel-side path filter `/app0/` и автоматически выставляет kernel-side `pid` filter на `pid` запущенной игры

## Разбор журнала Python-скриптом

Скрипт `analyze_pathlog.py` работает в двух режимах:
- построение Markdown-отчёта
- переименование файлов и папок

Скрипт:
- читает журнал pathlog, анализирует записи `open`
- собирает список файлов игры и сравнивает пути без учёта регистра
- умеет работать как с локальной папкой, так и по FTP

## Режим отчёта

Пример для локальной папки:

```bash
python3 pathlog-payload/analyze_pathlog.py \
  --report \
  pathlog-payload/PPSA12345.log \
  /games/PPSA12345
```

Пример для FTP:

```bash
python3 pathlog-payload/analyze_pathlog.py \
  --report \
  pathlog-payload/PPSA12345.log \
  ftp://user:pass@192.168.0.10:1337/user/app/PPSA12345
```

Если путь отчёта не задан, файл будет создан рядом с логом:

```text
<log>.report.md
```

Что попадает в отчёт:
- файлы, у которых отличается регистр имени
- каталоги, у которых отличается регистр имени
- конфликты регистра полных путей файлов в самом журнале
- конфликты регистра каталогов в самом журнале
- файлы, которые есть в папке игры, но не попали в журнал
- совпадающие файлы

Можно указать свой путь для отчёта:

```bash
python3 pathlog-payload/analyze_pathlog.py \
  --report \
  -o result.md \
  pathlog-payload/PPSA12345.log \
  /games/PPSA12345
```

## Режим переименования

Этот режим нужен, когда нужно привести реальные имена файлов и папок к регистру, который использует игра.

Сначала всегда рекомендуется запускать dry-run:

```bash
python3 pathlog-payload/analyze_pathlog.py \
  --rename-case \
  --dry-run \
  pathlog-payload/PPSA12345.log \
  /games/PPSA12345
```

Скрипт покажет, какие переименования он собирается сделать, но ничего не изменит.

После проверки можно запустить реальное переименование:

```bash
python3 pathlog-payload/analyze_pathlog.py \
  --rename-case \
  pathlog-payload/PPSA12345.log \
  /games/PPSA12345
```

То же самое работает и по FTP:

```bash
python3 pathlog-payload/analyze_pathlog.py \
  --rename-case \
  --dry-run \
  pathlog-payload/PPSA12345.log \
  ftp://user:pass@192.168.0.10:1337/user/app/PPSA12345
```

## Дополнительные параметры переименования

Можно отдельно задать регистр для файлов и папок, которые вообще не встретились в журнале.

Пример:

```bash
python3 pathlog-payload/analyze_pathlog.py \
  --rename-case \
  --dry-run \
  --unlogged-files-case lower \
  --unlogged-dirs-case lower \
  pathlog-payload/PPSA12345.log \
  /games/PPSA12345
```

Доступные варианты:
- `lower`
- `upper`

Это полезно, если ты хочешь не только исправить регистр по журналу, но и привести оставшиеся неиспользованные файлы и каталоги к единому стилю.

Ограничение:
- для `--unlogged-files-case` и `--unlogged-dirs-case` пути внутри `sce_module` и `sce_sys` специально игнорируются

## Параметры FTP

Если лог анализируется по FTP, можно передать параметры прямо в URL:

```text
ftp://user:pass@host:port/path/to/game
```

Либо переопределить их отдельными аргументами:
- `--ftp-user`
- `--ftp-pass`
- `--ftp-port`

Пример:

```bash
python3 pathlog-payload/analyze_pathlog.py \
  --report \
  --ftp-user user \
  --ftp-pass pass \
  --ftp-port 1337 \
  pathlog-payload/PPSA12345.log \
  ftp://192.168.0.10/user/app/PPSA12345
```

## Практические рекомендации

- Не запускай одновременно `pathlog_all.elf` и `pathlog_apps.elf`.
- Для игрового сценария обычно лучше `pathlog_apps.elf`.
- Перед реальным переименованием сначала делай `--dry-run`.
- Если нужно получить чистый лог одной игры, запусти `pathlog_apps.elf` до старта игры.
- Если нужно просто быстро проверить содержимое каталога, достаточно режима `--report`.

## Типичный сценарий работы

1. Загрузи специальную верисю kstuff-lite.
2. Запусти `pathlog_apps.elf`.
3. Запусти игру.
4. Дождись нужного момента в игре.
5. Останови monitor через `pathlog_stop.elf` или выйди из игры.
6. Возьми журнал из `/data/pathlog/<TITLE_ID>.log`.
7. Построй отчёт через `analyze_pathlog.py --report`.
8. При необходимости проверь план через `--rename-case --dry-run`.
9. Если план корректный, выполни реальное переименование.

## Если что-то не работает

Проверь по порядку:
- специальная версия kstuff-lite действительно загружена
- монитор был запущен до воспроизведения сценария
- журнал появился в `/data/pathlog/`
- для режима игр запущено приложение с кодом `CUSA*` или `PPSA*`
- при работе по FTP указан правильный путь к корню игры
- перед реальным rename сначала был проверен `--dry-run`
